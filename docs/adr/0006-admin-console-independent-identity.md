# Admin Console Owns an Independent Human Identity

The **Admin Console** authenticates human operators with an **Admin Account** and **Admin Session**, deliberately separate from the OAuth 2.1 client authorization the runtime already exposes. The existing OAuth surface authorizes MCP clients with tag-scoped tokens; it has no concept of a named human, password, or login. OAuth client tokens can never authenticate the Admin Console, and Admin Sessions can never act as MCP client tokens.

Passwords use Node's built-in `crypto.scrypt` rather than `bcrypt` or `argon2` because native addons do not bundle cleanly into the SEA single-file binary. Account records and server-side sessions persist through the existing `FileStorageService` idiom. Browser sessions are opaque, httpOnly, `SameSite=Strict` cookies backed by a TTL'd session store; CLI admin login stores only a scoped Admin Session reference and the runtime remains authoritative for expiry, revocation, password changes, account deletion, and disabled-admin behavior.

Accounts are bootstrapped outside the web surface: `1mcp admin create` for interactive operators, or environment-seeded first admin creation for headless and Docker boots. There is no unauthenticated first-run web account creation path, so a network-exposed runtime cannot be claimed by whoever reaches it first. The first release keeps one full-admin role with a stored `role` field for future RBAC, and it uses a fixed minimal password policy rather than configurable organization policy.
