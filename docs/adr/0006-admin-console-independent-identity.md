# Admin Console Owns an Independent Human Identity

The **Admin Console** authenticates human operators with an **Admin Account** and **Admin Session**, deliberately separate from the OAuth 2.1 client authorization the runtime already exposes. The existing OAuth surface authorizes MCP _clients_ with tag-scoped tokens; it has no concept of a named human, password, or login, and reusing it would conflate client authorization with operator identity. We therefore keep them as two unrelated trust systems: an OAuth client token can never authenticate the Admin Console, and an Admin Session can never act as an MCP client token.

Passwords are hashed with Node's built-in `crypto.scrypt` rather than `bcrypt`/`argon2`, because those are native addons that do not bundle cleanly into the SEA single-file binary; scrypt is salted, tunable, and ships with the runtime. Account records and sessions persist as JSON through the existing `FileStorageService` idiom (no database introduced). Sessions are opaque, httpOnly, `SameSite=Strict` cookies backed by a TTL'd session store, giving real server-side revocation and keeping tokens out of reach of XSS.

Accounts are bootstrapped outside the web surface — `1mcp admin create` for interactive operators, or an environment-seeded first admin for headless/Docker boots — never by an unauthenticated first-run web screen, so a network-exposed runtime cannot be claimed by whoever reaches it first. The Admin Console always requires a login (with a documented dev-only escape hatch). The account schema carries a `role` field from the start so future RBAC lands without a storage migration, even though the first release exposes a single full-admin role.
